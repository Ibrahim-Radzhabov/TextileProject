# Идеальный промпт для создания проекта уровня store-platform

Этот документ — результат анализа репозитория «с нуля». Он описывает **один цельный промпт**, по которому ИИ или команда могут воссоздать проект того же типа и уровня. Сфера бизнеса в промпте абстрактная (любой e-commerce / витрина), а не привязана к шторам или тюлю.

---

## Краткий анализ: что именно есть в проекте

- **Тип продукта:** конфиг-драйвенная, мультитенантная платформа витрины для интернет-магазина (один код — много магазинов).
- **Стек:** монорепо (pnpm workspaces + turbo): Next.js 14 App Router, FastAPI, общие пакеты (UI, shared-types, config-schema), клиентские JSON-конфиги в `clients/{CLIENT_ID}`.
- **Данные:** каталог и страницы задаются конфигами (catalog.json, pages.json и др.); заказы, идемпотентность, вебхуки, метрики PWA/избранного — в SQLite через один слой доступа (OrderStore).
- **Интеграции:** Stripe (checkout + webhooks), Telegram (уведомления), опционально; при отсутствии Stripe — graceful fallback (подтверждённый заказ без редиректа).
- **Витрина:** премиальный минималистичный дизайн, анимации (Framer Motion), темы из конфига (цвета, шрифты, тени), PWA (manifest, service worker, офлайн), SEO (metadata, sitemap, robots, JSON-LD).
- **Админка:** защита по токену (ADMIN_TOKEN), заказы, товары (CRUD), аналитика PWA и избранного, экспорт CSV, аудит вебхуков.
- **Качество и деплой:** валидация конфигов (Zod/JSON Schema), E2E (Playwright), Lighthouse PWA, smoke-тесты, runbook деплоя (Docker Compose, пост-деплой проверки).
- **Документация для агентов:** гайды по отчётам (UX, a11y, UI regression, стабильность, KPI), по безопасности (секреты, CORS, логи, бэкапы), по кэшу и edge (Nginx, Next.js), чек-листы handoff и по фичам.

Ниже — **идеальный промпт**, который всё это кодирует в виде задания.

---

## Идеальный промпт (копируй и подставляй нишу)

```
Создай с нуля конфиг-драйвенную мультитенантную платформу витрины для интернет-магазина в нише [ПОДСТАВЬ НИШУ: например «премиальный текстиль», «кофе и чай», «украшения ручной работы»]. Один код — много магазинов; каждый магазин определяется набором JSON-конфигов в каталоге clients/{CLIENT_ID}.

Требования к архитектуре и стеку:
- Монорепо (pnpm workspaces, turbo): приложение витрины (Next.js 14, App Router, RSC), бэкенд-API (FastAPI), общие пакеты: UI (общие React-компоненты), shared-types (TypeScript-типы Product, Order, конфиги), config-schema (Zod/JSON Schema для валидации конфигов клиента).
- Клиентские конфиги: shop (название, лого, валюта, контакты), theme (цвета, радиусы, тени, типографика, градиенты), catalog (товары, цены, медиа, теги), pages (страницы и блоки: hero, product-grid, rich-text, cta-strip и т.д.), seo (titleTemplate, defaultTitle, description, OG image), integrations (Stripe, Telegram). Все конфиги валидируются при инициализации и через скрипт validate-client.
- API читает конфиги по CLIENT_ID; заказы, идемпотентность checkout, вебхуки Stripe, метрики (PWA, избранное) хранятся в SQLite через единый слой доступа (миграции схемы внутри кода, без внешних миграционных движков на старте).

Функциональность витрины:
- Главная, каталог, карточка товара, избранное (синхронизация с бэкендом по sync_id), корзина (Zustand), checkout с созданием заказа и опциональным редиректом в Stripe; страница статуса заказа по order_id; статические страницы (privacy, offer, guides при наличии).
- Hero с вариантами (overlay/card, видео/изображение, параллакс и scroll reveal по желанию), блоки product-grid, rich-text, cta-strip; контент и блоки полностью из pages.json.
- Шапка: лого, навигация, поиск (по желанию — с раскрывающимся фильтром), иконки: главная, поиск, избранное (с бейджем), корзина (с бейджем); опционально — офф-канвас сайдбар слева (бургер/Close, overlay, блокировка скролла body) и выпадающее меню с «режимами» (пилюли) для сегментов каталога.
- Футер: колонки ссылок, контакты, соцсети, юридические ссылки; CTA из конфига.
- Темы: применение design tokens из theme.json (CSS-переменные), переключение вариантов темы по cookie (опционально).
- PWA: manifest, service worker, офлайн-fallback, опциональный баннер установки и метрики (prompt_shown, installed и т.д.) на бэкенд.
- SEO: metadata (title, description, OG, Twitter) на уровне layout и страниц; динамические sitemap и robots.txt; JSON-LD (Organization, WebSite, Product, BreadcrumbList) где уместно; canonical URL.

Бэкенд API:
- Роуты: storefront/config (агрегированный конфиг для клиента), catalog (продукты по slug/id, при необходимости CRUD для админки), cart/price (пересчёт корзины), checkout (создание заказа, идемпотентность по заголовку, при наличии Stripe — Checkout Session и redirect_url), orders (список, детали, смена статуса, экспорт CSV), webhooks/stripe (верификация подписи, дедупликация по event_id+client_id, обновление статуса заказа, Telegram при paid), favorites (sync get/put), metrics (приём событий PWA/избранного, админские списки и daily/export).
- Заказы: сохранение в SQLite (order_id, client_id, status, cart, customer, stripe_session_id, таймстемпы); аудит смены статусов; таблица идемпотентности checkout.
- CORS: только разрешённые origin из конфига/переменных окружения.

Админка:
- Защита всех маршрутов /admin/* по токену (ADMIN_TOKEN): страница логина, httpOnly cookie, middleware редиректа на /admin/login при отсутствии авторизации.
- Разделы: заказы (список с фильтрами по статусу/оплате, поиск, детали заказа, смена статуса), товары (список, создание/редактирование/удаление при наличии API CRUD), аналитика PWA (события, дневная сводка, экспорт CSV), аналитика избранного (события, экспорт). Аудит вебхуков Stripe по заказу/статусу обработки.

Качество и эксплуатация:
- E2E (Playwright): сценарии по витрине (главная, каталог, checkout success), API checkout, админка (логин → заказы → детали → логаут), PWA (manifest, sw, офлайн). Отдельные теги для визуальных и a11y-тестов.
- Lighthouse: скрипт проверки PWA (сборка, запуск, порог score и обязательные аудиты).
- Деплой: runbook (pre-deploy gate: build + e2e), теги релиза, чек-лист переменных (CLIENT_ID, FRONTEND_ORIGIN, STORE_API_URL, ADMIN_TOKEN, Stripe/Telegram), Docker Compose для API+Web, пост-деплой smoke по BASE_URL и API_URL.

Документация для ИИ-агентов:
- Гайд по отчётам: где хранить отчёты (UX-воронка, a11y, UI regression, стабильность, KPI), как именовать, как использовать шаблоны и связывать с тестами/кодом.
- Гайд по безопасности: секреты и env, защита админки, CORS, что не логировать, бэкапы БД, чек-лист до/после изменений.
- Гайд по кэшу и edge: что кэшировать (статику, HTML, API), что нет; пример конфига Nginx с кэшем и rate-limit; правила для Next.js (dynamic/revalidate); чек-лист при изменении кэша.
- Чек-листы: handoff для следующего агента (что сделано, что в работу), по крупным фичам (шапка, hero, док и т.д.).

Результат: после выполнения промпта должен получиться репозиторий, в котором можно добавить нового «клиента» копированием и правкой clients/demo → clients/new-client, выставить CLIENT_ID и переменные окружения, собрать и запустить витрину и API, прогнать E2E и деплой по runbook. Сфера [ниша] влияет только на примеры контента в конфигах и текстах, не на архитектуру и набор фич.
```

## Как использовать этот промпт

1. Подставь в промпт нишу вместо `[ПОДСТАВЬ НИШУ]` (например: «премиальный текстиль», «кофе и чай», «ювелирные украшения»).
2. Передай промпт ИИ или команде как единое ТЗ; при необходимости разбей на подзадачи по абзацам (архитектура → витрина → API → админка → качество → документация).
3. Для «облегчённой» версии можно выкинуть PWA, аналитику метрик, часть гайдов или мультитенантность — но тогда результат не будет того же уровня, что текущий store-platform.

Файл можно копировать в другие репозитории или дашборды промптов; при изменении самого проекта store-platform имеет смысл обновлять и этот документ, чтобы промпт оставался актуальным.
