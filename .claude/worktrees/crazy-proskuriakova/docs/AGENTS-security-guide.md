## Security Guide for AI Agents (Storefront)

**Audience:** агенты, настраивающие безопасность и доступы.  
**Goal:** обеспечить базовый, но надёжный уровень защиты витрины без лишнего усложнения.

---

### 1. Threat Model (что мы защищаем и от кого)

- **Поверхность:**
  - публичный фронт на Next.js (`apps/web`),
  - публичное API (`apps/api`), в том числе `/api/checkout`, `/api/catalog`, `/api/metrics/*`,
  - административная часть (`/admin` и `apps/api/routers/*` с `require_admin_token`),
  - файловая БД SQLite (`orders.sqlite3` и сопутствующие таблицы в `OrderStore`).
- **Угрозы, которые считаем реалистичными:**
  - перебор/утечка `ADMIN_TOKEN`,
  - спам/бот‑трафик на checkout и формы,
  - DoS лёгкого уровня (частые запросы к API, а не целенаправленная атака),
  - утечки секретов через логи, `.env`, CI.
- **Не цель:** защита от целевой APT, сложных 0‑day и т.п. (выходит за рамки проекта).

Агенты должны **сохранять этот баланс**: повышать безопасность без радикального усложнения инфраструктуры.

---

### 2. Secrets and Environment

**Файлы и места, где живут секреты:**

- `apps/api/.env` — используется `Settings` в `apps/api/config.py`.
- Секреты фронта/бэкенда в рантайме — через переменные окружения:
  - `ADMIN_TOKEN`
  - `CLIENT_ID`
  - `STORE_API_URL`
  - `NEXT_PUBLIC_STORE_API_URL`
  - `SITE_URL` / `NEXT_PUBLIC_SITE_URL`
  - Stripe/Telegram токены, если включены.

**Правила для агентов:**

1. **Никогда** не коммитить реальные секреты в Git.
   - Проверять изменения в `.env` и аналогах: если пользователь явно не попросил, **не добавлять**.
2. Для примеров использовать **фиктивные значения**:
   - `ADMIN_TOKEN=changeme_admin_token_very_long`,
   - `STRIPE_SECRET_KEY=sk_test_xxx`,
   - и т.д.
3. При правках `apps/api/config.py`:
   - не менять `env_prefix`, `env_file` без явного запроса;
   - не добавлять логирование секретов.

Если нужно добавить новый секрет:

1. Добавить поле в `Settings` (тип `Optional[str]` или специфичный тип).
2. Обновить `Config.env_file` только если пользователь об этом попросил.
3. В коде использовать только через `get_settings()`.

---

### 3. Admin Access and Authorization

**Как сейчас защищена админка:**

- В `apps/api/routers/admin_auth.py` (см. файл перед правками) реализована проверка `ADMIN_TOKEN`.
- В `apps/api/routers/metrics.py` и других админ‑роутерах используется `Depends(require_admin_token)`.

**Задачи агента:**

1. **Убедиться, что в проде `ADMIN_TOKEN` не дефолтный.**
   - В примерах документации обязательно писать, что `admin123` — только для локальной разработки.
2. Не расширять список эндпоинтов, доступных без авторизации, без необходимости.
3. При добавлении новых админ‑ручек:
   - всегда вешать `dependencies=[Depends(require_admin_token)]` или эквивалент;
   - не создавать параллельных «обходных» эндпоинтов.

**Дополнительные меры, которые агент может подготовить (но включать только по запросу):**

- IP‑фильтр перед `/admin` на уровне прокси (Nginx/Cloudflare).
- MFA или SSO через внешний провайдер (описать интеграцию, но не реализовывать без задачи).

---

### 4. CORS and Origins

Файл: `apps/api/config.py`, поле `cors_allow_origins` и функция `_normalize_origins`.

**Текущая логика:**

- CORS настраивается через список/строку, поддерживает несколько форматов.
- `frontend_origin` автоматически добавляется в `cors_allow_origins`, если его там нет.

**Рекомендации для агентов:**

1. Для прод‑окружения держать **минимальный набор origin’ов**:
   - основной домен витрины (`https://shop.example.com`),
   - при необходимости — админский домен.
2. Для dev/staging можно добавлять:
   - `http://127.0.0.1:3000`,
   - `http://localhost:3000`.
3. Никогда не выставлять `*` в проде без очень веской причины.

При изменениях:

- использовать `_normalize_origins`, не дублировать свою парсер‑логику;
- избегать логирования полного списка origin’ов, если там могут быть чувствительные значения.

---

### 5. Rate Limiting and Basic DoS Protection

Внутри FastAPI сейчас **нет полноценного rate‑limit’а**. Базовая защита предполагается через внешний слой (reverse‑proxy, Cloudflare, Ingress).

**Что должен сделать агент (пошагово, в отдельном конфиге):**

1. Подготовить пример Nginx/Cloudflare конфигурации (см. отдельный файл `AGENTS-caching-and-edge.md`):
   - лимиты запросов на IP/ключ для:
     - `/api/checkout*`,
     - `/api/metrics/*`,
     - `/admin/*`.
2. Не внедрять жёсткий rate‑limit прямо в код API без явного запроса пользователя.

Если пользователь попросил встроенный rate‑limit в API:

- использовать простые, проверенные библиотеки (например, `slowapi`) и описать:
  - какие ручки лимитируются,
  - какие коды ошибок и заголовки вернутся при превышении.

---

### 6. Logging and Sensitive Data

**Цели логирования:**

- понимать, когда API падает,
- видеть ошибки Stripe/checkout,
- иметь минимальную трассировку по инцидентам.

**Запрещено логировать:**

- полные тела запросов checkout с персональными данными,
- `ADMIN_TOKEN` и любые секреты,
- полные токены/ключи (Stripe, Telegram и пр.).

**Разрешено (и желательно) логировать:**

- статусы 5xx, текст ошибки без чувствительных полей,
- агрегированные метрики (количество ошибок, время ответа),
- ID заказа, но не полную customer payload.

При добавлении логов агент должен:

1. Явно фильтровать/маскировать поля с email, телефоном, адресом, токенами.
2. Использовать единый формат (JSON‑строки либо структурированные сообщения).

---

### 7. Backups and Data Safety (SQLite)

Текущая БД: SQLite (`orders.sqlite3`), путь берётся из:

- `Settings.order_db_path`, либо
- `apps/api/domain/order_store.py` → `data/orders.sqlite3` по умолчанию.

**Роль агента:**

1. **Никогда не удалять файл БД** в автоматических скриптах без явного подтверждения.
2. При изменении схемы:
   - использовать уже реализованные методы миграций в `OrderStore` (например, `_ensure_column`, `_ensure_webhook_audit_unique_constraint`);
   - избегать raw `DROP TABLE` / разрушительных операций.
3. В документации (например, `AGENTS-reporting-guide.md`) упоминать:
   - необходимость регулярных бэкапов файла БД (копия на S3/облако),
   - проверку целостности (банальный `pragma integrity_check` можно добавить в отдельный админ‑скрипт).

---

### 8. Checklist Before and After Security Changes

**Перед изменениями:**

- [ ] Прочитан соответствующий код (`apps/api/config.py`, `apps/api/domain/order_store.py`, нужные роутеры).
- [ ] Понято, какие окружения затрагиваются (dev/staging/prod).
- [ ] Секреты вносятся только через переменные окружения или `.env`, а не в код.

**После изменений:**

- [ ] Все админ‑ручки всё ещё защищены `require_admin_token` (или аналогом).
- [ ] CORS не ослаблен до `*` для прод‑доменов без причины.
- [ ] В логах нет утечки секретов или персональных данных.
- [ ] Бэкап БД либо не затронут, либо обновлён в документации.

Этот чек‑лист нужно проходить **каждый раз**, когда агент изменяет что‑то в области безопасности.
